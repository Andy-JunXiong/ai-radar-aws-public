# ADR-0008 Stage B Minimal Event Schema Design

## Goal

Define the smallest useful `SignalLifecycleEvent` shape for ADR-0008 Stage B.

The event model should support two consumers:

- per-signal Trajectory View
- future source-level lifecycle aggregation

This is a design artifact only. It does not implement storage, code, schema
migration, or hard enforcement.

## Non-Goals

- Do not implement `record_transition()`.
- Do not select final storage.
- Do not create a legal transition table.
- Do not refactor write paths.
- Do not backfill historical signals.
- Do not add hard rejections.
- Do not weaken Project Takeaway gates.
- Do not treat Reflection as factual evidence.
- Do not add a broad event ontology.

## Design Principles

1. Start with events that real write paths already need.
2. Prefer stable structured fields over free-text reasons for gate-relevant
   data.
3. Keep raw prompts, full external payloads, and sensitive source contents out
   of lifecycle events.
4. Represent legacy and inferred data honestly instead of backfilling it.
5. Add fields only when a known consumer needs them.
6. Preserve Project Takeaway, Reflection, and model provenance invariants from
   `docs/adr/INVARIANTS.md`.

## Minimal Event Envelope

Initial event records should use this envelope:

```json
{
  "schema_version": 1,
  "event_id": "sig_evt_...",
  "signal_id": "8db68a5936af4a22970a11ee9ca7cdfc",
  "event_type": "insight_generated",
  "event_time": "2026-05-22T00:00:00+00:00",
  "recorded_at": "2026-05-22T00:00:01+00:00",
  "actor": {
    "type": "system",
    "id": "ai-radar-backend"
  },
  "route": "/signals/generate-insight",
  "provenance_class": "direct",
  "source_ref": {
    "record_family": "signal",
    "record_id": "8db68a5936af4a22970a11ee9ca7cdfc"
  },
  "state": {
    "before": "pending",
    "after": "analyzed"
  },
  "support": {
    "verification_status": "partially_verified",
    "evidence_level": "medium",
    "allowed_downstream_actions": ["project_takeaway_candidate"],
    "blocked_downstream_actions": ["low_risk_action_candidate"],
    "evaluation_summary": "Verification metadata exists; action remains gated."
  },
  "project_ref": null,
  "model_provenance_ref": {
    "provider": "openai",
    "model_id": "gpt-5.5",
    "fingerprint_prefix": "57827a90"
  }
}
```

## Field Semantics

| Field | Required | Semantics |
|---|---:|---|
| `schema_version` | yes | Lifecycle event schema version. New Stage B events use `1`. |
| `event_id` | yes | Stable id for this event. Generated by lifecycle service. |
| `signal_id` | yes | Durable signal id or manual signal id. |
| `event_type` | yes | Stable lifecycle event type. |
| `event_time` | yes | Time the domain event happened. |
| `recorded_at` | yes | Time AI Radar recorded the lifecycle event. |
| `actor.type` | yes | `system`, `admin`, `import`, `legacy_adapter`, or future explicit actor type. |
| `actor.id` | yes | Stable actor identifier when available. |
| `route` | no | API route, service function, or job path that recorded the event. |
| `provenance_class` | yes | `direct`, `derived`, `inferred`, `legacy`, or `missing`. |
| `source_ref` | yes | Pointer to the owning source record without embedding raw payloads. |
| `state.before` | no | Previous known signal state when applicable. |
| `state.after` | no | Next known signal state when applicable. |
| `support` | no | Structured verification/gate snapshot when relevant. |
| `project_ref` | no | Project candidate/review reference when event concerns project fan-out. |
| `model_provenance_ref` | no | Compact reference to existing `produced_by_model` data; not evidence. |

## Provenance Classes

Use `provenance_class` to keep the UI honest:

- `direct`: recorded by the write path at the time of mutation
- `derived`: constructed from another durable product record, such as a
  ReviewRecord or CalibrationEvent
- `inferred`: reconstructed from current fields without authoritative event
  history
- `legacy`: older path or historical record without Stage B event recording
- `missing`: expected lifecycle evidence is absent

Rules:

- Only `direct` events are eligible for future hard-enforcement guarantees.
- `derived` events may support trajectory display but do not prove the original
  mutation path.
- `inferred`, `legacy`, and `missing` must be visibly labelled in the UI.

## Minimal Event Types

Start with a small set. Do not add event types until a write path or UI
consumer needs them.

| Event type | First source | Purpose |
|---|---|---|
| `signal_observed` | ingestion or legacy adapter | Signal entered AI Radar. |
| `insight_generated` | `/signals/generate-insight` | LLM or fallback judgment generated and stored. |
| `verification_completed` | insight generation / verification service | Verification metadata became available. |
| `signal_status_changed` | `/signals/update-status` | Operator or system status changed. |
| `workspace_completed` | `/signals/complete` | Workspace reflection/completion saved. |
| `project_candidate_created` | `add_signal_to_project_improvements()` | Project fan-out created a review candidate. |
| `project_review_attached` | ReviewRecord / CalibrationEvent services | Project-side review outcome attached to trajectory. |
| `override_applied` | override-confirm / override-action routes | Explicit manual override recorded. |

Possible future types such as `source_quality_aggregated`,
`reflection_promoted`, or `judgment_superseded` are intentionally excluded from
the initial set.

## Support Snapshot

`support` is optional, but required when the transition depends on verification
or downstream action gates.

Allowed keys:

```json
{
  "verification_status": "partially_verified",
  "evidence_level": "medium",
  "confidence_label": "high",
  "confidence_score": 0.75,
  "claim_support_summary": {
    "directly_supported": 2,
    "partially_supported": 1,
    "unsupported": 0,
    "contradicted": 0,
    "inferred": 1
  },
  "allowed_downstream_actions": ["project_takeaway_candidate"],
  "blocked_downstream_actions": ["low_risk_action_candidate"],
  "review_priority": "review_required",
  "evaluation_summary": "Short sanitized summary."
}
```

Rules:

- Do not store raw LLM prompts.
- Do not store full external source payloads.
- Do not store Reflection body text.
- Do not use `support` to override Project Takeaway gate logic.
- Store enough structured data for UI labels and future source-level
  aggregation.

## Project Reference

Project-side events should avoid pretending that Project ReviewRecords are
signal-owned mutation history.

Use `project_ref` for attachment:

```json
{
  "project_id": "ai_radar",
  "record_family": "project_review_record",
  "record_id": "review_...",
  "outcome": "watch"
}
```

Rules:

- Project review events can be `derived` lifecycle attachments before the
  project write paths are migrated.
- Ordinary Confirm and Action remain governed by Project Takeaway gates.
- Override events must point to the explicit override path and note/expected
  outcome metadata.

## Model Provenance Reference

Lifecycle events may reference model provenance, but they must not duplicate
the full `produced_by_model` object unless a consumer needs that snapshot.

Initial reference shape:

```json
{
  "provider": "openai",
  "model_id": "gpt-5.5",
  "fingerprint_prefix": "57827a90",
  "schema_version": 1
}
```

Rules:

- Model provenance remains judgment provenance, not evidence.
- Full provenance remains on the judgment-producing record.
- Legacy/v0 provenance should be labelled as coverage state, not failure.

## Reflection Exclusion

Reflection is excluded from automatic lifecycle enforcement.

No initial event type should record ordinary Reflection browsing, sync, or
polish as factual verification history.

If a future product path promotes Reflection into Signal, Knowledge, or Project
review, that promotion should use a specific event type and verification
requirements designed at that time.

## Conceptual Service Interface

The eventual service can be designed around these conceptual operations:

```text
record_transition(event_draft) -> SignalLifecycleEvent
list_signal_events(signal_id) -> list[SignalLifecycleEvent]
derive_legacy_trajectory(signal_id) -> list[SignalLifecycleEvent]
summarize_lifecycle_by_source(source_id, window) -> summary
```

This document does not decide module path, storage engine, persistence format,
or API routes.

## First Soft-Recording Candidates

Recommended implementation order:

1. `/signals/generate-insight`
   - records `insight_generated`
   - records or co-records `verification_completed` when verification metadata
     exists
2. `/signals/complete`
   - records `workspace_completed`
   - records `project_candidate_created` when project fan-out writes candidates
3. Project ReviewRecord / CalibrationEvent reads
   - expose `project_review_attached` as `derived` events before project-side
     write paths are migrated

Do not start with hard enforcement.

## Test Expectations

First implementation tests should prove:

- direct event creation includes required envelope fields
- verification-dependent transitions include `support`
- Project Takeaway blocked actions remain blocked
- override events require explicit override metadata
- legacy/inferred events are labelled as non-authoritative
- Reflection content is not stored as event evidence
- model provenance references do not create trust or gate eligibility

## Open Questions

- Should `verification_completed` be a separate event or part of
  `insight_generated` for the first implementation?
- Should manual upload session ids be normalized before or inside lifecycle
  event recording?
- Does source-level aggregation need source id on every event in the first
  implementation, or can it be derived from `source_ref` for the first slice?
- Should event ids be deterministic hashes or generated ids?
- Where should soft-recorded local/dev lifecycle events persist before any
  production storage decision?

## Current Decision

This design is sufficient to start an implementation planning slice.

The next slice should still be narrow:

- pick one first soft-recording path
- implement only the lifecycle service shape needed by that path
- keep legacy/inferred trajectory display honest
- do not enable hard enforcement
